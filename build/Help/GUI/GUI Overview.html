GUI Overview


SuperCollider provides for the possibiltiy for using different gui kits, and also provides syntax for transparently writing kit and platform independent code. Two main classes control this mechnanism: the factory abstraction class,  GUI, and the subclasses of  ViewRedirecter. The default gui kit is cocoa. 


Kits and Syntax


Kit-dependant gui classes  usually have a prefix, fore example,  to make a window, the cocao kit uses SCWindow  or swing kit uses JSCWindow.  It is generally discouraged to directly use the kit-dependant classes directly. Instead,  there are basically two techniques for writing kit-independent code: using GUI and using  the subclasses of  ViewRedirecter.


For a table of equvalent gui classes, see GUI Classes.


Using the GUI class directly, you call a GUI class method with the same name as the view class base name.  

 

w=GUI.window.new("Test Window").front; // note you must call .new explicitly  with this method

w.postln // w now caontains the class appropriate to the gui kit


To simplify the syntax, you can use the subclasses of ViewRedirecter, and  simply use the class name without any prefix. Behind the scenes, the following technique actually automatically redirects to the appropriate GUI class method.

 

w=Window("Test Window").front; // Window is a subclass of ViewRedirecter

w.postln // w now caontains the class appropriate to the gui kit


Depending on the curent GUI.scheme two examples above are equivalent to writing, for example:


SCWindow("Test Window").front; // cocoa version

or

JSCWindow("Test Window").front; // swing version

For information o switching gui kits, see the documentation for GUI.



Basic Usage: Windows, Containers and Decoraters


The following documentation will use the concise  ViewRedirecter  syntax mentioned above.

For a complete list of gui classes  , see GUI Classes.


(

var w, b;

// GUI.window returns the window class for the current kit

w = Window( "my name is... panel", Rect( 128, 64, 340, 360 ));



w.view.decorator = FlowLayout( w.view.bounds ); //comment this out for no decorator

w.view.background = Color( 0.6, 0.8, 0.8 );


32.do({ arg i;

// the same is true for all other widgets

// ; here GUI.button returns the button class for the current kit

b = Button( w, Rect( rrand( 20, 300 ), rrand( 20, 300 ), 75, 24 ));

b.states = [[ "Start " ++ i, Color.black, Color.rand ],

  [ "Stop " ++ i, Color.white, Color.red ]];

});


w.front;

)


In the above example (Basic Usage), first a Window was created. A Window automatically has a container, an instance of CompositeView (actually a kit dependent subclass, topView or scrollTopView).  A container can have a decorater, which automatically places the contents in a predetermined order. In this case a FlowLayout, which is the only one that currently exists. This is optional however. Try commenting out the decorator in the example above.


Most gui objects or subclasses of View. All subclasses of View are created with the same beginning arguments, parent and bounds, for example, Slider(parent, bounds).


The parent is a container view of some kind, and the bounds are and instance of Rect or a Point (for width and height only). Using a Point is convenient if you are using a decorator for the postioning.


(

w=Window.new.front;

Slider(w, Rect (50,50,250,20));

)


You can easily nest containers,  decorators and views to make complex layouts, and you can use colors to visually separate them:

(

w=Window.new.front;

v=CompositeView(w, Rect(5,5,190,390));

v.background= Color.rand;

v.decorator=FlowLayout(v.bounds);

y=CompositeView(w, Rect(205,5,190,390));

y.background= Color.rand;

y.decorator=FlowLayout(y.bounds);

14.do{

Slider(v, 180@20); // Points used, since the layout is handled by a decorator.

Slider(y, 180@20);

};

)

Helper Methods in GUI


*stringBounds( string, font )


Returns a Rect object describing the bounds occupied by the given string if it was painted using the given font. Note that this method is asynchronous in SwingOSC, hence it is advised to use it inside a Routine. Example:


(

{

var text = "Test", bounds, font, fonts, rect = Rect.new, total = Rect.new;

fonts = GUI.font.availableFonts;

w = GUI.window.new( "String Bounds", resizable: false );

w.view.background = Color.blue;

10.do({

font = GUI.font.new( fonts.choose, exprand( 6, 36 ));

bounds = GUI.stringBounds( text, font );

rect.set( rect.right, rect.bottom, bounds.width + 4, bounds.height + 2 );

GUI.staticText.new( w, rect )

.font_( font ).align_( \center ).string_( text ).background_( Color.white );

total = total.union( rect );

});

w.bounds = total.moveTo( 200, 200 );

w.front;

}.fork( AppClock );

)




Accessing "Plus-GUI"methods


"Plus-GUI" methods are methods added to other classes such as String or Server that provide GUI functionality for those classes. These methods use the current GUI kit implementation as returned by GUI.current .


.inspect 

Examples for Inspectors:

Server.default.inspect;

Server.inspect;


.browse

Examples for Browsers:

UGen.browse; // ClassBrowser

SynthDescLib( \myLib ).read.browse;


.makeWindow

Examples for Server:

if( s.window.notNil, { s.window.close });

s.makeWindow;

s.scope;

Extending GUI Kits


GUI Kits can be extended with custom classes by using their respective .put methods:


GUI.get( \cocoa ).put( \myText, SCStaticText );

GUI.get( \swing ).put( \myText, JSCStaticText );


GUI.cocoa;

GUI.swing;

(

w = GUI.window.new;

GUI.myText.new( w, w.view.bounds.insetBy( 20, 20 )).string_( "schoko" ).background_( Color.red );

w.front;

)


If you intend to add extensions from within your own classes upon class library initialization time, the preferred way is to do this in the startup process:


MyGUIExtension {

*initClass {

StartUp.add({

var scheme;

scheme = GUI.get( \cocoa );

if( scheme.notNil, { scheme.put( \myText, SCStaticText )});

scheme = GUI.get( \swing );

if( scheme.notNil, { scheme.put( \myText, JSCStaticText )});

});

}

}
